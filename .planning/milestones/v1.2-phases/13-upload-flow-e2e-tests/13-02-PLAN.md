**Test describe: "Upload Validation and Error Handling"**

All tests navigate to `/demo?tool=java-verification`.

**Invalid file type tests:**

1. **"rejects .txt file with error message"**
   - Upload `INVALID_TXT_PATH` via `uploadPage.uploadFile()`
   - Client-side react-dropzone rejects based on `accept` config (only `application/zip` + `.zip`)
   - Wait briefly for rejection to process (use `page.waitForTimeout(500)` since rejection is synchronous)
   - Assert `uploadPage.uploadError` is visible OR `uploadPage.uploadSuccess` is NOT visible
   - If upload-error is visible, assert error text contains "ZIP" or "invalid" (case-insensitive)
   - Note: react-dropzone may show error via `uploadError` with "Only ZIP files are accepted" message

2. **"rejects .jpg file with error message"**
   - Upload `INVALID_JPG_PATH`
   - Same assertion pattern as test 1
   - Assert error is visible with message about ZIP/invalid type

3. **"rejects file with no extension"**
   - Upload `NO_EXTENSION_PATH`
   - Assert rejection — react-dropzone checks both MIME type and extension
   - Assert error visible or success NOT visible

**Oversized file test:**

4. **"rejects oversized file with size error message"**
   - Use `createOversizedFile(101 * 1024 * 1024)` to create a 101MB temp file (just over 100MB limit)
   - Upload the oversized file
   - Wait for error state
   - Assert `uploadPage.uploadError` is visible
   - Assert error text contains "too large" or "100MB" (case-insensitive)
   - Clean up: delete the oversized temp file using `cleanupTempFile()`
   - Mark test as `test.slow()` since creating large file takes time

**Error recovery tests:**

5. **"Try Again button resets error state to idle"**
   - Upload invalid file, wait for error
   - Click Try Again button
   - Assert `uploadPage.uploadStatus` is visible (idle state)
   - Assert `uploadPage.uploadError` is not visible

6. **"error then recovery: upload invalid file, see error, then upload valid file and see success"**
   - Upload `INVALID_TXT_PATH`, wait for error/rejection
   - If error state with Try Again: click Try Again first to reset
   - Upload `SAMPLE_ZIP_PATH`
   - Wait for upload success
   - Assert success indicator visible
   - Assert project ID is present
   - This proves the full error-recovery-success cycle

**Empty/corrupted ZIP test:**

7. **"empty ZIP file uploads but shows zero files extracted"**
   - Upload `EMPTY_ZIP_PATH`
   - Two possible outcomes depending on implementation:
     a. Client accepts it (it IS a valid .zip), server processes it: success with 0 files
     b. Server rejects it as invalid: error state
   - Assert either: success text contains "0 files" OR error is shown
   - Use `Promise.race` or check both states with short timeouts

**Important implementation notes:**
- react-dropzone in UploadZone.tsx uses `accept: { 'application/zip': ['.zip'] }` and `maxSize: MAX_UPLOAD_SIZE`
- Client-side rejections trigger `fileRejections` callback which sets `rejectionError` state
- Client rejection errors show in `upload-error` div with specific messages: "Only ZIP files are accepted" (type), "File too large (max 100MB)" (size)
- The error display includes a "Try Again" button (actually a "Try Again" text in the Button, triggered by `reset()`)
- Actually per the code: error state shows "Upload Failed" heading, then the error message, then "Try Again" button

For oversized file test: since creating 101MB file is expensive, use `test.slow()` and clean up in `test.afterEach` or inline cleanup.
