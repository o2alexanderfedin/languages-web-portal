### Phase 21 Design Decision Required

`cs-fv verify` takes a single FILE argument (not a directory). The portal execution service passes `--input <projectDir>`. Phase 21's wrapper must choose:
- **Option A (recommended):** Iterate `.cs` files in `--input <dir>` and call `dotnet /usr/local/lib/cs-fv/cs-fv.dll verify <file>` per file — correct formal verification semantics
- **Option B (wrong):** Call `cs-fv check <dir>` — only validates contract syntax, not formal verification

## Tech Debt

### Phase 20 Deferred Items

**1. Image size deviation (1546 MB vs 800 MB — human-approved)**
- Primary drivers: NuGet/Roslyn cache 333 MB, cs-fv DLL 75 MB, solver binaries 66 MB, dotnet-runtime-8.0 ~100-200 MB
- Functional requirements unaffected — all DOCKER-01 through DOCKER-04 verified
- Optimization path: NuGet cache pruning (remove arch-specific packages) could reduce size
- Severity: Low (acknowledged deviation, no production impact for demo portal)

**2. Java-builder pre-built jar workaround**
- `java-builder` stage uses COPY of pre-built `cli-1.1.0-jar-with-dependencies.jar` instead of Maven build
- Root cause: `FormulaAdapter.adaptForIncremental()` called in `PooledSolverProcess.java` but not implemented
- Risk: Jar tracked in git — if java-fv source diverges, Docker build silently uses stale jar
- C# FV is unaffected — uses `dotnet-builder` and `solver-builder` exclusively
- Severity: Medium (maintenance risk, no current functional impact)
