  <action>
Run the following commands to inspect the Docker image contents. The image tag is `hupyy-languages-web-portal` (or as named in the local Docker daemon — check with `docker images | grep hupyy` or `docker images | grep languages`).

Step 1 — Find the image:
```
docker images | grep -E "hupyy|languages"
```

Step 2 — List DLLs in the cs-fv publish directory:
```
docker run --rm <image> ls /usr/local/lib/cs-fv/*.dll
```
Expected: should include `CsFv.Contracts.dll` (because dotnet publish of CsFv.Cli includes all ProjectReference DLLs in the publish output).

Step 3 — Verify the cs-fv verify invocation (does it accept a .cs file and read .csproj?):
```
docker run --rm <image> dotnet /usr/local/lib/cs-fv/cs-fv.dll verify --help 2>&1 | head -20
```
This reveals whether `verify` takes a `.cs` file path, a `.csproj` path, or a directory — which determines whether the `.csproj` `<Reference>` HintPath is used.

Step 4 — If CsFv.Contracts.dll is NOT present:
Inspect the publish output to find where the DLL is:
```
docker run --rm <image> find /usr/local/lib/cs-fv/ -name "*.dll" | sort
```

Record findings. The .csproj HintPath in Plan 02 must point to the exact confirmed path.

**Fallback if CsFv.Contracts.dll is absent from publish output:**
cs-fv loads its own contract types as part of its Roslyn analysis pipeline — the `using CsFv.Contracts;` in example `.cs` files may be resolved by cs-fv internally (not via MSBuild <Reference>). In that case, the .csproj needs NO `<Reference>` to CsFv.Contracts.dll — just `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` and the other MSBuild properties. Verify this by testing:
```
docker run --rm <image> bash -c 'echo "using CsFv.Contracts; public class T { [Requires(\"x > 0\")] public int F(int x) => x; }" > /tmp/t.cs && dotnet /usr/local/lib/cs-fv/cs-fv.dll verify /tmp/t.cs'
```
If this exits 0 with verification output, cs-fv resolves CsFv.Contracts internally and the `<Reference>` is optional (only needed for IDE support).
