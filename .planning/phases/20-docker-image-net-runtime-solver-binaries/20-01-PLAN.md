<task type="auto">
  <name>Task 2: Add dotnet-builder stage to Dockerfile</name>
  <files>Dockerfile</files>
  <action>
Insert the dotnet-builder stage into the Dockerfile AFTER the solver-builder stage and BEFORE the production stage. The stage header must be:

```
# Stage 4: .NET Builder
FROM --platform=$BUILDPLATFORM mcr.microsoft.com/dotnet/sdk:10.0-noble AS dotnet-builder
ARG TARGETARCH
```

The `--platform=$BUILDPLATFORM` flag is REQUIRED here — it prevents Docker from QEMU-emulating the .NET SDK for the target architecture, which would make the build extremely slow on Apple Silicon.

The stage must:
1. Copy cs-fv source from build context (monorepo root):
   `COPY cs-fv/ /build/cs-fv/`
   Set WORKDIR to /build/cs-fv

2. Set NuGet global packages location BEFORE the dotnet command:
   `ENV NUGET_PACKAGES=/nuget-packages`
   This ensures dotnet restore/publish writes packages to /nuget-packages, which we COPY to production.

3. Build and publish the cs-fv CLI:
   ```
   RUN dotnet publish src/CsFv.Cli/CsFv.Cli.csproj \
         --configuration Release \
         --framework net8.0 \
         --output /publish/cs-fv \
         --no-self-contained \
         -p:MinVerSkip=true
   ```
   - `--no-self-contained`: framework-dependent deployment; production stage installs dotnet-runtime-8.0
   - `-p:MinVerSkip=true`: MinVer reads git tags; Docker COPY does not preserve .git history — skip to avoid build failure
   - `--framework net8.0`: cs-fv targets net8.0 per Directory.Build.props
   - Output DLL will be at /publish/cs-fv/cs-fv.dll (AssemblyName=cs-fv in CsFv.Cli.csproj)
   - NuGet packages will be at /nuget-packages/ (populated automatically by the implicit restore during publish)

Update the production stage comment to "Stage 5: Production".
  </action>
