**Example 1: Bank Account Records** (simplest - no explicit ACSL contracts, verification infers from code)
- README.md: First line after heading = "Records with compact constructor invariants — automatic verification of value constraints"
- Account.java: A `record Account(String owner, int balance)` with compact constructor that validates `balance >= 0` and `owner != null`. Add a `deposit(int amount)` method that returns new Account (amount must be > 0). Add a `withdraw(int amount)` method returning new Account (amount must be > 0 and <= balance). Java FV infers SMT constraints from these runtime checks.
- Transaction.java: A `record Transaction(Account from, Account to, int amount)` with compact constructor validating amount > 0. Add a static method `transfer(Account from, Account to, int amount)` that performs a withdrawal and deposit, returning the two new accounts. Demonstrate that the transferred amounts are consistent (from.balance + to.balance remains constant).

**Example 2: Shape Matching** (medium - pattern matching with type guards and null handling)
- README.md: First line after heading = "Pattern matching with type patterns, guards, and null safety verification"
- Shape.java: A sealed interface `Shape` with records `Circle(double radius)`, `Rectangle(double width, double height)`, and `Triangle(double a, double b, double c)`. Circle validates radius > 0, Rectangle validates width > 0 and height > 0, Triangle validates all sides > 0 and triangle inequality (a + b > c, etc.).
- ShapeCalculator.java: A class with `static double area(Shape shape)` using pattern matching switch expression with type patterns. Each case extracts fields and computes area. Also include `static String classify(Shape shape)` that uses guards: e.g., `case Circle c when c.radius() > 100 -> "large circle"`. Verification proves exhaustiveness of the sealed type matching.
- NullableShape.java: A class demonstrating null pattern matching. Method `static String describe(Shape shape)` that handles `case null -> "no shape"` along with all Shape subtypes. Verification ensures null safety.

**Example 3: Payment Types** (complex - sealed types with exhaustiveness + intentional verification failure)
- README.md: First line after heading = "Sealed payment types with exhaustiveness checking — includes a deliberate verification failure"
- PaymentMethod.java: A sealed interface `PaymentMethod` permits `CreditCard`, `BankTransfer`, `Crypto`. Record `CreditCard(String number, int expiryMonth, int expiryYear)` validates month 1-12 and year >= 2024. Record `BankTransfer(String iban, String bic)` validates non-null non-empty. Record `Crypto(String walletAddress, String currency)` validates non-null.
- PaymentProcessor.java: A class with `static double calculateFee(PaymentMethod method, double amount)` using switch expression over sealed type. CreditCard = 2.9% + 0.30, BankTransfer = 0.5%, Crypto = 1.0%. Add `static boolean isValid(PaymentMethod method)` that checks business rules. Verification proves exhaustiveness and fee calculations are non-negative.
- UnsafeRefund.java: **INTENTIONAL FAILURE**. A class with `static double processRefund(double originalAmount, double refundAmount)` that DOES NOT validate refundAmount <= originalAmount, allowing negative balance. Add a comment: `// WARNING: This method intentionally lacks proper validation`. Also include a method that accesses array without bounds check. Java FV should detect these violations and produce "VERIFICATION FAILED" output.

**Java code style notes:**
- Use modern Java syntax (records, sealed interfaces, pattern matching switch)
- No package declarations (flat file structure for portal demo simplicity)
- No main methods needed (Java FV verifies code properties, not runtime behavior)
- Keep files 30-60 lines each for readability in the portal console
- Use clear variable names and brief comments explaining what verification catches
