- All tests run only on desktop project: `test.skip(({ isMobile }) => isMobile, 'Execution tests run on desktop only')`
- Use `test.describe.configure({ mode: 'serial' })` since these are expensive Docker tests and should not run concurrently

**Common helper:** Create a helper function `loadExampleAndRun(page: Page, exampleName: string)` that:
1. Navigates to `/demo?tool=java-verification`
2. Creates DemoPage instance
3. Waits for exampleSelector to be visible
4. Calls `loadExample(exampleName)` to load the example
5. Clicks execute button
6. Returns the DemoPage instance for assertions

**Tests — Happy path (successful verification):**

1. **'bank-account-records example executes and shows VERIFIED'**
   - Load and run bank-account-records
   - Wait for streaming: assert a loading/progress indicator appears (the "Running..." text on execute button OR the "Streaming output..." text, or the spinning connection state badge)
   - Wait for console output to appear (data-testid="console-output" has text content)
   - Wait for execution to complete: use `page.waitForFunction` checking console output contains 'completed' or 'exit code' (same pattern as DemoPage.waitForExecutionComplete but with 180s timeout)
   - Assert console output text contains at least one of: 'VERIFIED', 'verified', 'precondition', or verification-related keywords
   - Assert the execution result status badge shows 'COMPLETED' (the green badge in ExecutionPanel)

2. **'shape-matching example executes successfully'**
   - Load and run shape-matching
   - Wait for execution complete (180s timeout)
   - Assert console output contains verification-related text

3. **'payment-types example shows verification failures for UnsafeRefund.java'**
   - Load and run payment-types
   - Wait for execution complete (180s timeout)
   - Assert console output contains 'FAILED' or 'failed' or 'VERIFICATION FAILED'
   - Check for specific failure mode keywords from UnsafeRefund.java's 5 intentional failures. Check for as many as the tool reports — at minimum check for presence of failure-related output. Look for keywords like: 'null', 'division', 'overflow', 'bounds', 'refund', 'unsafe', 'ArithmeticException', 'NullPointerException', 'ArrayIndexOutOfBoundsException'
   - Note: the exact output depends on what Java FV reports. Assert that at least some failure indicators are present.

**Tests — Streaming verification:**

4. **'streaming output shows early markers before final result'**
   - Load and run bank-account-records
   - Use `page.waitForFunction` to check that console-output has ANY text content (early marker — something appeared before completion)
   - Store a snapshot of current console text
   - Then wait for final completion
   - Assert the final console text is longer than the early snapshot (more output was streamed after the early marker)
   - This proves streaming occurred (output arrived incrementally, not all at once)

5. **'auto-scroll behavior during streaming'**
   - Load and run bank-account-records (or any example)
   - Wait for console output to have some content
   - Use `page.evaluate` to check `scrollTop + clientHeight >= scrollHeight - 10` on the console-output element (scrolled to bottom within 10px tolerance)
   - Wait for execution to complete
   - Check scroll position is still near bottom after more output streamed

6. **'loading indicator visible during execution'**
   - Load an example, click execute
   - Assert execute button contains text matching /Running/i (the streaming state shows "Running...")
   - Assert a connection state badge is visible (the "CONNECTING" or "CONNECTED" badge shown during streaming in ExecutionPanel)
   - Wait for execution to complete
   - Assert button no longer shows "Running..." — it should show the tool name again or "Run Again"

**Tests — Output file tree (E2E-04):**

7. **'output file tree appears after successful execution'**
   - Load and run bank-account-records
   - Wait for execution complete
   - Assert the "Output Files" heading (h3 with text "Output Files") becomes visible
   - Assert `data-testid="output-panel"` is visible
   - Assert the output panel contains at least one tree node (look for `role="treeitem"` elements inside output-panel, or text that matches file names)

8. **'output file tree contains verification artifacts'**
   - Load and run bank-account-records (or reuse state if serial)
   - Wait for execution complete
   - Wait for output-panel to be visible
   - Check the output panel text for generated file names — verification tools typically produce reports or annotated source files
   - Assert at least one file is listed in the tree

All tests should use `test.slow()` marker since they involve real Docker execution.
